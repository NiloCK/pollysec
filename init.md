# Polly Wants a Second to Think About That

## Implementation Spec v1.0 — Bracket Matching with Looped Transformers and Architectural Registers

---

## 1. Overview

We train four small transformer variants on bracket matching and measure whether (a) looping enables depth generalisation and (b) explicit architectural registers enable learned cross-iteration control state.

**Single task:** Given a string of `(` and `)`, output `balanced` or `unbalanced`.

**Training depths:** 1–8. **Test depths:** 1–16.

**Four model variants**, all ~200K parameters (hidden dim 64):

| Variant | Looping | Registers |
|---|---|---|
| V1 — Vanilla | No | No |
| V2 — Vanilla + Reg | No | Yes |
| V3 — Looped | Yes (4 iterations) | No |
| V4 — Looped + Reg | Yes (4 iterations) | Yes |

---

## 2. Vocabulary and Tokenisation

Character-level. No subword tokeniser needed.

| Token | ID |
|---|---|
| `PAD` | 0 |
| `(` | 1 |
| `)` | 2 |
| `CLS` | 3 |

The input sequence is: `CLS` followed by the bracket string, right-padded with `PAD` to a fixed length.

Classification is read from the `CLS` position at the final layer. The output head projects that position's hidden state to 2 logits (balanced / unbalanced). Standard cross-entropy loss.

**Maximum sequence length:** 34 tokens (1 `CLS` + up to 32 bracket characters + 1 `PAD` minimum). Depth-8 balanced brackets can be at most 16 characters; depth-16 test inputs can be at most 32 characters. This gives comfortable headroom.

**Why CLS at the start rather than end:** The model processes left-to-right with causal attention. Placing CLS first means it attends to nothing initially but accumulates information as later positions attend back to it through the layers. However — for this task, **bidirectional attention is actually appropriate** since we're doing classification, not generation. The model should see the full bracket string before deciding.

**Decision: use bidirectional (full) attention**, not causal masking. CLS at position 0, bracket string following, padded. Attention mask zeros out PAD positions. Classification from the CLS position's final hidden state.

## 3. Data Generation

All data generated deterministically from fixed seeds.

### 3.1 Generating bracket strings

For a given depth `d`:

1. **Balanced examples.** Generated by a recursive procedure that produces strings with maximum nesting depth exactly `d`. Approach: generate a random Dyck word (balanced bracket string) and filter/regenerate until its max depth equals `d`. Alternatively, use a constructive method:
   - Start with the depth-`d` kernel: `d` opens followed by `d` closes → `(((...)))`
   - Randomly insert additional matched pairs at legal positions to vary the structure
   - This ensures depth is exactly `d` while varying string length and shape

2. **Unbalanced examples.** Take a balanced string of depth `d` and apply one random corruption:
   - Flip one bracket (open→close or close→open)
   - Delete one bracket
   - Insert one extra bracket
   This keeps the unbalanced examples structurally similar to balanced ones, preventing the model from using superficial cues (e.g. odd string length).

### 3.2 Splits

| Split | Depths | Seed | Size |
|---|---|---|---|
| Train | 1–8, uniform | 42 | 200,000 examples |
| Val | 1–8, uniform | 43 | 10,000 examples |
| Test-ID | 1–8, uniform | 44 | 10,000 examples |
| Test-OOD | 9–16, uniform | 45 | 10,000 examples (1,250 per depth) |

50/50 balanced/unbalanced at every depth.

All four model variants train on the identical Train split in the identical order (same batching seed). Val is used for early stopping / checkpoint selection. Test-ID and Test-OOD are touched only for final evaluation.

### 3.3 Implementation

Data generation is a standalone script (`data.py`) that writes the four splits to disk as JSON lines:

```json
{"input": "(()())", "label": 1, "depth": 3}
```

Label: 1 = balanced, 0 = unbalanced.

Generate once, reuse for all runs. Check into the repo alongside the code.

## 4. Model Architecture

### 4.1 Common components (all four variants)

**Embedding.** Learned token embeddings, dim 64. Learned positional embeddings for positions 0–33. Sum of token + position embedding as input to layer 1.

**Transformer layer.** Each layer is a standard pre-norm transformer block:
1. RMSNorm → Multi-head self-attention (4 heads, head dim 16) → residual add
2. RMSNorm → Feedforward (up-project to 256 via linear, SiLU activation, down-project to 64 via linear) → residual add

Full (bidirectional) attention, with PAD positions masked out.

**Output head.** Take the hidden state at the CLS position (position 0) after the final layer. Apply RMSNorm → Linear(64, 2). Two logits: [unbalanced, balanced]. Cross-entropy loss against the label.

**Number of layers:** 6 (for all variants; looped variants tie and reuse these same 6 layers).

### 4.2 Variant-specific details

#### V1 — Vanilla

Standard 6-layer transformer. Single forward pass.

#### V2 — Vanilla + Registers

Same 6-layer transformer. Single forward pass. Plus: register vector and register update MLP (described in §4.3). The register is updated once per layer (6 updates total during the single pass). The register is readable by each layer but there is no cross-iteration carry because there are no iterations.

Implementation: the register is updated after each transformer layer, and its contents are injected into the next layer's input (see §4.3 for injection mechanism). This makes V2 a fair comparison — the register machinery is active, it just doesn't get the benefit of looping.

#### V3 — Looped

6 transformer layers with **tied weights**. The full 6-layer stack is applied `T` times (max `T=4` during training). Each iteration takes the previous iteration's output hidden states as input and runs all 6 layers again.

**Exit gate:** A small linear layer (64 → 1 → sigmoid) applied to the mean-pooled hidden state after each iteration. Outputs a probability `p_exit` ∈ [0, 1].

During training: all `T=4` iterations always execute. The loss is a weighted sum across iterations (see §5). The exit gate is trained but doesn't actually truncate computation during training.

During inference: after each iteration, if `p_exit > threshold` (default 0.8), stop and use the current hidden state for classification. Otherwise continue to the next iteration. Always stop at `T_max`.

For test-time extrapolation experiments, `T_max` can be increased beyond 4 to see if the model benefits from additional iterations.

#### V4 — Looped + Registers

Same as V3, plus the register mechanism. The register vector is initialized to zeros before iteration 1. After each complete pass through the 6-layer stack, the register is updated and carried to the next iteration. Within each iteration, the register is also updated per-layer (same as V2), so the register gets 6 intra-iteration updates plus the carry across iterations.

### 4.3 Register mechanism (V2 and V4)

**Register vector:** `r` ∈ ℝ^8. (Sized smaller than first proposed — 8 dims is ~12% of the hidden state width. Large enough to encode a counter and a few flags; small enough that it's clearly a side-channel, not a second hidden state.)

**Register update MLP:** After each transformer layer, the register is updated:

```
r_input = concat(h_cls, r)    # h_cls is the CLS position hidden state, dim 64+8 = 72
r_mid = SiLU(Linear(72, 32))  # up-project
r_new = Linear(32, 8)         # down-project back to register size
r = r + r_new                 # residual connection on the register itself
```

This reads from the CLS hidden state (a summary of sequence-level information) and the previous register state, and writes back to the register. The residual connection helps with gradient flow across many updates.

**Register injection into transformer layers:** Before each transformer layer, the register is broadcast to all positions and added to the hidden state via a learned linear projection:

```
h = h + Linear(8, 64)(r).unsqueeze(seq_dim)  # project r to hidden dim, add to all positions
```

This is lightweight (8×64 = 512 parameters) and lets every position read the register contents.

**Key constraint:** The register does NOT feed into the output head. The output head reads only from the CLS hidden state after the final layer. The register influences the output only indirectly, by modifying the hidden states during processing.

**For V4 specifically — cross-iteration carry:** After iteration `ℓ` completes (all 6 layers done), the register `r` is carried unchanged to the start of iteration `ℓ+1`. It is NOT reset. This is the mechanism that enables cross-iteration communication.

## 5. Training

### 5.1 Loss function

**For V1 and V2 (non-looped):** Standard cross-entropy on the output logits.

**For V3 and V4 (looped):** Loss is computed at every iteration. The output head is applied to the CLS hidden state after each iteration's 6 layers, producing logits at each iteration. Final loss:

```
L = Σ_{ℓ=1}^{T} w_ℓ * CrossEntropy(logits_ℓ, label)
```

Where `w_ℓ = ℓ / sum(1..T)` — later iterations weighted more heavily. For T=4: weights are [0.1, 0.2, 0.3, 0.4]. This encourages all iterations to produce good outputs but prioritises the later ones.

**Exit gate regularisation (V3 and V4 only):** An entropy bonus on the exit gate distribution encourages the model to explore different exit depths rather than collapsing to always using all 4 or always exiting at 1:

```
L_gate = -λ * H(p_exit distribution across iterations)
```

Where λ = 0.01 (small relative to task loss). The exit distribution is constructed from the per-iteration exit probabilities as a categorical: probability of exiting at iteration ℓ = p_exit_ℓ * Π_{k<ℓ}(1 - p_exit_k).

Total training loss for looped variants: `L_task + L_gate`.

### 5.2 Optimiser and schedule

- AdamW, lr = 3e-4, weight decay = 0.01, betas = (0.9, 0.999)
- Linear warmup over first 1,000 steps, then cosine decay to 1e-5
- Batch size: 128
- Training steps: 30,000 (may increase if val loss hasn't plateaued; monitor and decide)
- Gradient clipping: max norm 1.0

### 5.3 Checkpointing and early stopping

Save a checkpoint every 2,000 steps. Track validation accuracy. Use the checkpoint with best validation accuracy for final evaluation.

### 5.4 Seeds and repetitions

Three random seeds per variant: 100, 200, 300. These seeds control:
- Model weight initialisation
- Dataloader shuffling order (but the data itself is fixed)

Data generation seeds (42–45) are fixed and shared across all runs.

Total training runs: 4 variants × 3 seeds = 12 runs.

## 6. Evaluation

### 6.1 Core metrics

**Accuracy vs. depth.** For each model, compute accuracy separately at each depth 1–16. Plot as a line chart with depth on the x-axis. One line per variant, averaged across 3 seeds with error bars (±1 std).

This is the hero plot. We expect:
- V1 (vanilla): high accuracy depths 1–8, rapid collapse at 9+
- V2 (vanilla + reg): similar to V1 (registers don't help without looping)
- V3 (looped): high accuracy 1–8, better than V1 at 9–12, eventual degradation
- V4 (looped + reg): high accuracy 1–8, best performance at 9–16

**Compute allocation.** For V3 and V4: plot average exit iteration vs. input depth. We expect an increasing trend — the model learns to use more iterations for deeper inputs.

### 6.2 Probing analysis

Freeze the trained V4 model. Extract:
- The register vector `r` after each iteration (4 snapshots per input)
- The CLS hidden state `h_cls` after each iteration (4 snapshots per input)

Train linear probes (logistic regression / linear regression) on these representations to predict:
1. **Current iteration number** (1–4) — can the probe tell which pass we're on?
2. **Input nesting depth** (1–16) — does the representation encode problem difficulty?
3. **Current max unmatched depth at CLS** — does the register track the "stack pointer"?
4. **Final answer** (balanced / unbalanced) — how early in the iterations does the answer become decodable?

Train each probe separately on register vs. CLS hidden state. If the register encodes control state more cleanly, probes on the 8-dim register should match or beat probes on the 64-dim hidden state for items 1–3, while the hidden state should be better for item 4 (since it feeds the actual output).

### 6.3 Ablation

Take the trained V4 model. At test time, intervene:

**Register zeroing:** After each iteration, set `r = 0` before the next iteration. The register still gets updated within each iteration but doesn't carry across iterations. Compare accuracy vs depth against unmodified V4.

**Register freezing:** After iteration 1, freeze `r` to its iteration-1 value for all subsequent iterations. The register carries some state but can't update. Compare accuracy vs depth.

**Register noise:** Add Gaussian noise (varying σ) to the register between iterations. Measure accuracy degradation as a function of noise magnitude. If the register carries important information, even small noise should hurt extrapolation performance.

Expected result: all three interventions damage extrapolation (depth 9–16) more than in-distribution (depth 1–8), confirming the register's role in enabling recursive depth generalisation.

3. `train.py` — training loop with checkpointing and validation logging. Verify: V1 overfits a tiny dataset (100 examples) to 100% accuracy.

### Phase 2 — Core models (Day 3–5)
4. Add V2 (vanilla + registers) to `model.py`. Verify: forward pass, register updates happen, shapes correct.
5. Add V3 (looped) to `model.py`. Verify: weight tying works, multi-iteration loss computes, exit gate produces probabilities.
6. Add V4 (looped + registers) to `model.py`. Verify: register carries across iterations, shapes correct.
7. Sanity check: train all four on a tiny dataset. All should reach ~100% accuracy on depths 1–4.

### Phase 3 — Full training runs (Day 6–8)
8. Run all 12 training runs (4 variants × 3 seeds). On Kaggle: fits in ~6–10 hours total. On laptop CPU: spread across 2–3 overnight runs.
9. Produce accuracy vs. depth plots. Check whether the core result holds.

If V4 doesn't outperform V3, the register mechanism isn't helping — consider adjusting register size, injection method, or update rule before running probing.

11. `probe.py` — train probes on frozen V4 representations.
12. `ablate.py` — register zeroing/freezing/noise experiments.

## 10. Success Criteria

**Strong result (publish-worthy):**
- V4 accuracy at depth 12+ is >80% while V1 is <30%
- V3 shows intermediate performance (better than V1, worse than V4)
- V2 ≈ V1 (registers without looping don't help)
- Probes on the register recover iteration count and nesting depth with >90% accuracy
- Register ablation collapses extrapolation while preserving in-distribution performance

**Moderate result (still publishable with caveats):**
- V3 and V4 both beat V1 at extrapolation, but V4 doesn't clearly beat V3
- Probes show the register encodes *something*, but it's noisy
- This would suggest looping is the main mechanism and registers are a nice-to-have

**Null result (still informative, publish as negative result):**
- No variant generalises beyond training depths
- Would suggest bracket matching at this scale doesn't require recursive strategies — the model finds non-recursive shortcuts
- Valuable because it constrains the design space

## 12. Future Work (V2 of the experiment)

Deferred from this spec, to be revisited if V1 results are positive:

- **Parenthesis scoring task** — tests memoisation, not just counting. Registers would need to function as a value stack.
- **Multi-task training** — bracket matching + scoring simultaneously. Tests whether registers specialise per task.
- **Learned jump targets** — instead of looping the full 6-layer stack, let the model choose which layer to re-enter at. More granular than full-stack looping.
- **Scaling to pretrained models** — retrofit registers onto an existing small LM and test on naturalistic recursive tasks.
